Production sysadmin, agency edition.

Maintaining live WordPress sites for marketing-agency clients. The work that pulled me from "web dev" toward infrastructure.

takeaway

What this taught me.

Production servers don't care how clean your code is. They care whether the cron actually ran, whether the permissions actually let the web user write to that directory, whether DNS actually propagated. Same skills, applied at small scale. The cloud version is the same problems with more zeros — and a control plane.

01
the_setup

SSH into the server first, fix the bug second.

Most of my "web dev" job at the agency starts with SSH. The frontend changes happen on a laptop; the verifying-it-actually-works part happens on the server. That single fact is what made the systems coursework feel directly applicable instead of abstract.

02
incident_01

When the bug isn't in the code.

A client reported outbound email failing. Nothing in the WordPress logs, nothing in the PHP errors. The fix was in DNS — missing MX record, broken SPF. Twenty minutes once you know what to look for; an entire day if you've never read a zone file. CS 356 paid for itself that afternoon.

03
incident_02

Cron jobs and the principle of least privilege.

The previous setup ran maintenance scripts as root and gave the web user permission to overwrite any file under /var/www. Setting up cron correctly under www-data and locking down the upload directory permissions was a small change with a real security delta. Boring work, important work.

04
why_this_matters

The cloud version is the same problems, bigger.

Everything on this page — DNS, cron, permissions, the principle of least privilege — has a direct analog in AWS. Route 53 is DNS. EventBridge is cron. IAM is permissions. The reason the cert path makes sense is that the substrate is already familiar; what changes is the control plane.
